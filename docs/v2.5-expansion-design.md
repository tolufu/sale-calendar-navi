## 0. v2.5+ のゴールと非ゴール

将来拡張（ビックカメラ等の家電量販店、Yahoo!ショッピング/Qoo10、B2B別ライン、ASP商品フィード、
店舗CSVフィード、公開共有ページ、ランキング、海外マーケット）を**安全に積み増せる土台**を設計する。
本書は「どう拡張するか」の設計図であり、**v2.5 で全部を実装するという意味ではない**（§9 で優先順位、
§10 で最小スコープを提案）。

### ゴール
- 現状の `merchantId` / `merchants` マスタ設計が EC 追加に耐えるかを点検し、**不足を補う拡張項目**を定義する。
- EC 別 feature flag、ASP フィード Provider、CSV インポート、公開共有、ランキングの**型と境界**を設計する。
- 法務・権利・プライバシーの**ガードレール**（提携前ECの情報を扱わない／断定表現禁止／個人情報を共有しない）を
  各機能の設計に最初から織り込む。

### 非ゴール（やらない / 禁止）
- **提携前 EC の商品情報・画像・価格を勝手に取得しない**（スクレイピング・無断 API 利用の禁止は据え置き）。
  新 EC は「公式に許可された取得元」または「手入力」のみ。許可が取れるまでは**カレンダー/メタ情報のみ**扱う。
- **B2B 系を既存（個人向け）ユーザー体験に無理に混ぜない**（§8 で別モード/別ラインとして分離）。
- 「最安」「最安保証」等の**断定・保証表現**、公式サービスと**誤認されるロゴ・表現**（ランキングでも同様）。
- 公開共有での**個人情報・userId・メモ・内部URLの露出**（§6）。
- API キー・ASP トークン等の**クライアント露出 / リポジトリへのコミット**。

## 1. 現状の merchantId 設計の EC 追加耐性（点検結果）

### 1.1 結論

**データ層は EC 追加に耐える。ただし「マスタにレコードを足すだけ」を阻むコード内ハードコード辞書が 3 つある。**
v2.5 の前提作業として、この 3 つを**マスタ駆動**へ寄せるのが最優先（破壊的でない小改修）。

### 1.2 耐えている点（良い設計・維持する）

| 観点 | 現状 | 評価 |
| --- | --- | --- |
| EC 識別子 | `Merchant.merchantId: string`（enum / union literal に固定していない） | ◎ 追加に強い |
| 参照側 | `SaleEvent` / `Offer` / `WishItem` / `PriceCandidate` / `ViewHistory` が全て `merchantId: string` 参照 | ◎ 型の破壊なしで追加可 |
| マスタ項目 | `name` / `colorToken` / `placeholderKey` / `affiliate` / `isActive` / `sortOrder` を保持 | ○ 表示・並び・有効化はマスタ駆動 |
| EC フィルタ / 色分け | カレンダーのフィルタ・並びはマスタ参照（`src/data/merchants.ts`） | ○ レコード追加で反映 |
| アフィリエイト | `affiliate{provider,enabled}` の設定駆動（v1.1 は楽天のみ） | ○ 設定で on/off |

### 1.3 EC 追加を阻むハードコード（v2.5 で解消すべき）

| # | 箇所 | 問題 | 解消方針 |
| --- | --- | --- | --- |
| H1 | `src/lib/utils/merchant.ts` `merchantToneClasses` | `colorToken → Tailwind クラス`が**コード内の固定 Record**。新 EC 色を足すと毎回コード編集が必要。 | マスタ（または色トークン定義表）駆動にし、未知トークンは中立配色へフォールバック（既に fallback はある）。Tailwind の動的クラスは**安全な静的マップ + safelist**で扱う（§2.3）。 |
| H2 | `src/lib/utils/merchant.ts` `detectMerchantIdFromUrl` の `knownHosts` | `merchantId → ホスト配列`が**コード内の固定 Record**。新 EC のURL自動判定を足すたびにコード編集。 | マスタに `urlHosts: string[]` を追加し、**マスタから突合**（§2.2）。コードのフォールバック表は段階的に縮小。 |
| H3 | `src/lib/utils/affiliate.ts` の `provider !== "rakuten"` 直書き | アフィリエイト変換が**楽天専用に分岐**。新 ASP/EC の変換を足すと if が増える。 | `provider` をキーにした **AffiliateProvider レジストリ**へ（§4 と共通化）。楽天変換は既存関数のまま 1 provider として登録。 |

> 補足: `extractAmazonAsin`（amazon 固有）や placeholder SVG 資産は EC 固有で当然コード/資産が要る。
> これらは「追加コストが妥当な固有処理」であり H1〜H3 とは別。新 EC ごとに**プレースホルダー資産の用意は必須**
> （外部画像を使わない方針のため）。

### 1.4 slug ルーティングの整理（任意・小）

- 現状 `app/calendar/[merchantSlug]/page.tsx` は `merchantSlug` を **`merchantId` と同一視**して突合している
  （`item.merchantId === merchantSlug`）。当面は merchantId をそのまま slug に使えるが、将来 URL 体裁を変えたく
  なった場合に備え、マスタに **任意の `slug?: string`** を持たせ「`slug ?? merchantId`」で解決する余地を残す
  （v2.5 必須ではない）。

## 2. merchants マスタの拡張項目

既存フィールドは**削除しない**。**任意フィールドの追加のみ**（後方互換）。`merchantId: string` 参照は不変。

### 2.1 拡張後の `Merchant` 型（設計案・`src/lib/repositories/types.ts`）

```ts
export type MerchantKind = "marketplace" | "retailer" | "b2b"; // 既定 "marketplace"

export type MerchantDataSource =
  | "manual"        // 手入力のみ（提携前 EC の既定。商品情報は扱わない）
  | "official-api"  // 公式 API 許可済み（例: 楽天）
  | "asp-feed"      // ASP 商品フィード経由（§4）
  | "csv-import";   // 店舗 CSV 取り込み（§5）

export type Merchant = {
  merchantId: string;
  name: string;
  colorToken: string;
  placeholderKey: string;
  affiliate: { provider: string; enabled: boolean } | null;
  isActive: boolean;
  sortOrder: number;

  // ===== v2.5 追加（すべて任意・既定値あり） =====
  kind?: MerchantKind;             // 既定 "marketplace"。B2B 分離・表示分類に使う（§8）
  line?: "consumer" | "business";  // 既定 "consumer"。プロダクトライン分離（§8）
  slug?: string;                   // 任意。URL 用。未指定なら merchantId（§1.4）
  urlHosts?: string[];             // URL→merchant 自動判定用（§1.3 H2）。例 ["amazon.co.jp","amzn.asia"]
  dataSource?: MerchantDataSource; // 既定 "manual"。商品情報の取得元を明示（許可の有無を型で表現）
  partnership?: {                  // 提携状態。未提携 EC を誤って自動取得しないためのガード
    status: "none" | "pending" | "approved";
    approvedScopes?: ("image" | "price" | "affiliate" | "feed")[]; // 許可された範囲のみ扱える
  };
  region?: "JP" | "global";        // 既定 "JP"。海外マーケット対応の足場（§9）。通貨は別途 currency 検討
  notes?: string;                  // 運用メモ（非表示）
};
```

### 2.2 `urlHosts` のマスタ駆動化（H2 の解消）

- `detectMerchantIdFromUrl(url, merchants)` を **`merchant.urlHosts` から突合**するよう変更（純粋関数のまま）。
- コード内 `knownHosts` は当面**フォールバック**として残し、マスタに `urlHosts` がある EC はマスタ優先。
- 既存 amazon/rakuten は `urlHosts` をマスタに記載すれば挙動同一（回帰テストで担保）。

### 2.3 `colorToken` のマスタ駆動化（H1 の解消・Tailwind 安全策）

- Tailwind は**動的クラス文字列を生成しない**（PurgeCSS で消える）原則を守るため、`colorToken → 静的クラス`の
  対応は**1 か所の静的マップ**に集約し、`tailwind.config.ts` の **safelist** に必要クラスを登録する。
- 新 EC 色の追加手順を「①マスタに `colorToken` ②静的マップに 1 行 ③ safelist に登録」に**定型化**して文書化。
  完全な「レコード追加のみ」にはならない（Tailwind の制約）が、**未知トークンは中立配色へ安全に fallback**する
  ので**追加し忘れても壊れない**ことを保証する（既存 fallback を維持）。
- 代替案: CSS 変数（`--merchant-color`）+ インラインスタイルにすれば safelist 不要にできる。v2.5 では
  既存実装の影響を抑えるため**静的マップ + safelist を既定**とし、CSS 変数化は任意の改善とする。

### 2.4 マスタの所在

- 現状どおり `src/data/merchants.ts`（ローカルシード）+ Firestore `merchants/{merchantId}`。
- 追加項目は**シード JSON とリポジトリ層の両方**で後方互換に読めること。`MerchantRepository` interface は変えない。

## 3. EC 別 feature flag

新 EC を「カレンダーには出すが、商品情報連携はまだ」のように**段階的に有効化**するためのフラグ設計。
**「カレンダー表示の有効化」と「商品情報の取得許可」を分離**するのが要点（提携前 EC を安全に扱う）。

### 3.1 フラグの 2 階層

1. **マスタ駆動の能力（capability）**: 各 EC が「何をしてよいか」を `merchant.dataSource` /
   `merchant.partnership.approvedScopes` /` merchant.affiliate` で表現。これは**データであり、提携実態に対応**する。
2. **環境/運用トグル（env flag）**: その能力を「この環境で実際に動かすか」を env で切替（鍵の有無とも連動）。

### 3.2 機能可否の判定（純粋関数で集約・設計案）

```ts
// src/lib/merchants/capabilities.ts（新・純粋関数）
export type MerchantCapability =
  | "calendar"        // カレンダー/セール予定の表示（提携不要。手入力の予定情報）
  | "manualWishlist"  // 手入力での欲しいもの登録（提携不要）
  | "image"           // 商品画像の表示（official-api/asp-feed かつ approvedScopes に "image"）
  | "priceSuggestion" // 参考価格の取得（approvedScopes に "price"）
  | "affiliate"       // アフィリエイト変換（affiliate.enabled かつ approvedScopes に "affiliate"）
  | "feedImport";     // フィード/CSV 取り込み（asp-feed/csv-import かつ approvedScopes に "feed"）

export function getMerchantCapabilities(
  merchant: Merchant,
  env: { rakutenApi: boolean; aspFeed: boolean; csvImport: boolean } // env flag の集約
): Set<MerchantCapability>;
```

- 規則: `calendar` / `manualWishlist` は**提携不要で常に可**（手入力・予定情報のみ）。
  `image` / `priceSuggestion` / `affiliate` / `feedImport` は **`partnership.status === "approved"` かつ
  `approvedScopes` に該当**し、**かつ対応する env flag が true** のときだけ付与。
- **既定は最小権限**（`partnership` 未設定 = `none` 扱い → 商品情報系は付かない）。提携前 EC を足しても
  「カレンダーに出るだけ」で安全。

### 3.3 env flag（`.env.example` 追記・キー名のみ）

```bash
# ===== v2.5+: 機能トグル（サーバ判定が真の可否。フラグだけでは有効化しない） =====
# ASP 商品フィード（サーバ専用・鍵は別途 ASP ごとに）
ENABLE_ASP_FEED=
# 店舗 CSV インポート（管理者導線のみ）
ENABLE_CSV_IMPORT=
# 公開共有ページ
NEXT_PUBLIC_ENABLE_PUBLIC_SHARE=
# ランキング表示
NEXT_PUBLIC_ENABLE_RANKING=
```

- v2 の `NEXT_PUBLIC_ENABLE_RAKUTEN_API` と同じ流儀: **クライアント露出してよいトグルだけ `NEXT_PUBLIC_`**、
  鍵やサーバ判定が要るものは `NEXT_PUBLIC_` を付けない。
- **フラグ true でも鍵/提携が無ければ動かない**（v2 の `isRakutenApiConfigured` と同じ二重ガード）。

## 4. ASP フィード Provider 設計

ASP（アフィリエイト・サービス・プロバイダ）の**商品フィードを、規約で許可された範囲でのみ**取り込む設計。
**「提携・許可された ASP の、許可されたフィードのみ」**が大前提（無断の商品 DB 構築はしない）。

### 4.1 レイヤ構成（新設 `src/lib/integrations/asp/`）

```
src/lib/integrations/asp/
  types.ts        … AspFeedItem / AspProvider interface / 設定型
  registry.ts     … providerId → AspProvider の登録表（楽天/将来ASPを差し込む）
  config.ts       … isAspConfigured(providerId)（サーバ専用 env 読取）
  providers/
    <aspName>.ts  … 各 ASP の mapper（純粋関数）+ client（サーバ専用 fetch）
```

### 4.2 Provider interface（設計案）

```ts
// 取得元に依存しない正規化済みアイテム
export type AspFeedItem = {
  providerId: string;        // ASP 識別
  merchantId: string;        // 紐づく EC（マスタに存在し approved であること）
  externalId: string;        // ASP 側の商品ID（冪等な upsert キー）
  title: string;
  productUrl: string;        // 通常URL
  affiliateUrl: string | null;
  imageUrl: string | null;   // 許可ホスト検証を通った1枚。NG は null（§5 と共通の isAllowedImageUrl）
  price: number | null;      // 参考価格（手入力 breakdown を上書きしない）
  currency: "JPY";           // v2.5 は JPY 固定（海外は将来）
  fetchedAt: string;         // ISO
};

export interface AspProvider {
  readonly providerId: string;
  isConfigured(): boolean;                              // サーバ env 判定
  search(input: { keyword: string; hits?: number }): Promise<
    | { ok: true; items: AspFeedItem[]; fetchedAt: string }
    | { ok: false; reason: "not-configured" | "rate-limited" | "upstream-error" | "invalid-input" }
  >;
}
```

### 4.3 設計原則（v2 楽天 adapter と一貫させる）

- **クライアントから ASP を直接叩かない**。必ず Route Handler 経由（`/api/asp/[providerId]/search` など）。鍵を秘匿。
- `mapper` は**ネットワーク非依存の純粋関数**でテスト（生 JSON → 正規化 → **許可ホスト外画像は null**）。
- **取得対象の EC は `partnership.status==="approved"` かつ `approvedScopes` に `"feed"`** のものだけ。
  未提携 EC を providerId 経由で取得しようとしたら**拒否**（型 + ランタイム両方でガード）。
- 取り込んだ `price` は v2 の `apiSuggestion` と同様**参考値**。`breakdown`（手入力）を自動上書きしない。
- レート制御・短時間キャッシュ・タイムアウトを Route 側に実装（v2 §3.4 を踏襲）。
- **既存の楽天 API（v2）も将来この registry に 1 provider として寄せられる**ようにしておく（無理に今統合しない）。

## 5. CSV インポートの安全設計

店舗が自分の商品/セール情報を CSV で登録する将来機能。**信頼できない入力**として扱い、防御的に設計する。
v2.5 では**運用者/店舗管理導線（限定）**を想定し、一般ユーザーには開かない。

### 5.1 脅威と対策

| 脅威 | 対策 |
| --- | --- |
| CSV インジェクション（`=`,`+`,`-`,`@`,Tab,CR 始まりのセル → Excel で数式実行） | 取込時に該当先頭文字を**エスケープ/除去**して保存。出力時も同様。**実行可能な値として解釈しない**。 |
| 巨大ファイル / 行数爆発（DoS） | **サイズ上限・行数上限・列数上限**を設定。ストリーム処理 + 上限超過で中断。 |
| 不正/未提携 merchantId の混入 | `merchantId` は**マスタに存在し `approved` な EC のみ許可**。未知は行ごと reject。 |
| 危険 URL（`javascript:` 等）/ 許可外画像ホスト | URL は **https のみ**許可、`imageUrl` は §5 共通 `isAllowedImageUrl` を通す。NG は null/reject。 |
| 文字コード/区切り崩れ | UTF-8 前提・BOM 除去・厳格パーサ（quote/escape 準拠）。曖昧な行は reject しログ。 |
| 個人情報/機密の混入 | 取込スキーマを**ホワイトリスト列**に限定（想定外の列は破棄）。価格・在庫の**断定保存はしない**。 |
| なりすまし（他店の商品を登録） | アップロード主体の**認可**（その merchant を扱える運用者か）を Route 側で必須化。監査ログ。 |

### 5.2 取込パイプライン（設計案）

```
アップロード(管理導線)
  → サイズ/拡張子/MIME 検証
  → パース（厳格・上限つき）
  → 行ごと正規化（ホワイトリスト列・型変換・CSV インジェクション無害化）
  → 検証（merchantId 存在+approved / URL https / 画像許可ホスト / 必須列）
  → ドライラン結果プレビュー（件数・reject 理由）   ← 人間が確認してから確定
  → 確定で upsert（externalId で冪等）
```

- **純粋関数で「1 行 → 正規化結果 or reject 理由」**を実装しテスト（インジェクション無害化・上限・許可判定）。
- **ドライラン（プレビュー）必須**: いきなり本番反映しない。reject 行と理由を返す。
- 取り込んだ価格は **`sourceType:"imported"`**（既存 `Offer.sourceType` に `"imported"` あり）として区別表示し、
  参考値・店舗申告である旨を明記（断定表現禁止）。

### 5.3 配置

```
src/lib/import/csv/
  schema.ts      … 許可列ホワイトリスト + 型
  sanitize.ts    … CSV インジェクション無害化・URL/画像検証（純粋関数・テスト対象）
  parse.ts       … 上限つき厳格パース → 行配列
  validate.ts    … 行→ {ok:正規化} | {reject:理由}（純粋関数）
```
- API/管理導線（`app/api/admin/import/...`）は**認可必須**で別タスク。コアの純粋関数を先に固める。

## 6. 公開共有ページのプライバシー設計

「欲しいもの」リスト等を**公開URLで共有**する将来機能。既定は**非公開**。共有は**明示的オプトイン**かつ
**個人情報・内部情報を一切含めない**。既存 `share.ts` の「機密を共有文面に入れない」方針を Web 共有へ拡張する。

### 6.1 原則

- **既定は非公開。** 公開は項目を選んでの**明示オプトイン**のみ。ワンクリックで全公開しない。
- **公開ペイロードは別物（公開ビューモデル）**。`WishItem` をそのまま出さず、**ホワイトリスト射影**する。
- 露出禁止: `userId` / メールや通知設定 / 自由メモ（`note`/`priceMemo`/`actualPriceMemo`）/ `affiliateUrl` の
  追跡パラメータ / 内部 ID / `lastCheckedAt` 等の行動が読み取れる時刻 / アップロード元情報。
- 公開しても**ユーザー本人特定につながらない**こと（共有スナップショットに ID を埋めない）。

### 6.2 公開トークンとアクセス

```ts
export type PublicShare = {
  shareId: string;          // ランダム不可推測トークン（列挙不可・十分な長さ）
  // userId は保存するが公開ペイロードには出さない（サーバ側の所有確認用）
  createdAt: string;
  expiresAt: string | null; // 任意の有効期限。既定で期限つきを推奨
  revoked: boolean;         // いつでも失効可能
  payload: PublicShareItem[]; // ホワイトリスト射影済みスナップショット
};

export type PublicShareItem = {
  title: string;
  merchantName: string;     // 表示名のみ（色は中立 or マスタ色）
  desiredPrice: number | null;
  // 画像は許可ホスト検証済み URL のみ（§5）。無ければ placeholder
  imageUrl: string | null;
};
```

- **トークンは推測不可・列挙不可**。URL を知る人だけがアクセス（unlisted）。検索エンジンには **`noindex`**、
  `sitemap`/`robots` に**含めない**。
- **失効・期限**をユーザーが管理できる（`revoked`/`expiresAt`）。失効後は 404/410。
- 公開は**スナップショット**（生データへの直結ではない）。後からメモを足しても公開分は増えない設計。
- 共有ページに**広告/アフィリエイト**を出す場合も、追跡で本人特定できる形にしない。規約表記は守る。

### 6.3 法務・表現

- 「最安」「最安保証」等の断定をしない（個人ページでも同様）。価格は本人申告の手入力値である旨を明記。
- 第三者の商品画像は**許可ホストのみ**（§5）。提携前 EC の画像は出さない。

## 7. ランキングで避けるべき誤認表現

ランキング（人気/注目のセール・商品等）を出す場合の**表現ガードレール**。誤認・断定・優良誤認を避ける。

### 7.1 禁止・回避する表現

| 区分 | 避ける表現 | 理由 / 代替 |
| --- | --- | --- |
| 価格断定 | 「最安」「最安値」「最安保証」「底値」 | 自動取得していない/保証できない。→「ユーザー登録が多い」等**事実ベース**に。 |
| 品質断定 | 「ベスト」「No.1」「絶対お得」「買い確定」 | 主観/保証。→「注目度が高い」「登録数が多い」。 |
| 公式誤認 | EC 公式ランキングのロゴ/名称/UI 模倣、「Amazon 公式ランキング」等 | 公式と誤認させない。**自社集計である旨**を明記。 |
| 順位の根拠不明 | 根拠を示さない順位 | **集計基準・期間・母数を明示**（例「当サイトの登録数・直近30日」）。 |
| ステマ/PR 不明示 | 広告/アフィリンクを通常順位と混同 | アフィリ/PR は**明示**。順位とアフィリ有無を**混同させない**。 |
| 個人特定 | 「○○さんが登録」等 | 集計は**匿名・集約のみ**（個票を出さない）。 |

### 7.2 設計上のガード

- ランキングの**並び基準は集計可能な事実**（登録数・閲覧数・対象セール件数など）に限定し、
  **基準・期間・母数のラベルを必ず併記**（UI 文言を定数化しテスト）。
- 価格に基づくランキングは、**手入力/参考値である旨**を必ず注記し「最安」系の語を**コード/コピーから機械的に排除**
  （禁止語リントを検討）。
- 提携前 EC の商品はランキング対象に**含めない**（`approved` のみ）。

## 8. B2B 系の分離（別モード/別ライン）

**禁止事項「B2B を既存ユーザー体験に無理に混ぜない」**を設計で担保する。

- マスタの `merchant.line: "consumer" | "business"` と `kind: "b2b"` で**データから分離**。
- 既定の個人向け画面は **`line === "consumer"` のみ**を対象にフィルタ（B2B は混ざらない）。
- B2B は**別モード/別ルート**（例 `/* business 名前空間 */`）または**別ライン（別ビルド/別サブドメイン）**として
  検討。共通化するのは**コア型・リポジトリ抽象・状態 UI まで**で、画面・コピー・導線は分ける。
- v2.5 では「**分離できる型の足場（`line`/`kind`）を入れるだけ**」に留め、B2B 画面自体は実装しない（§9）。

## 9. 実装優先順位

「安全な土台 → 価値の高い拡張 → 重い/審査の要る拡張」の順。各段は**前段が無くても既存が壊れない**こと。

| 優先 | 項目 | 理由 / 依存 | 規模感 |
| --- | --- | --- | --- |
| **P0** | §1.3 ハードコード解消（H1 colorToken / H2 urlHosts / H3 affiliate registry）＋ §2 マスタ拡張（任意フィールド） | 以降すべての土台。後方互換の小改修。EC 追加コストを下げる。 | 小〜中 |
| **P0** | §3 capability 判定の純粋関数 + env flag（提携前 EC を安全に足せる二重ガード） | 新 EC を「カレンダーだけ」で安全に出すための前提。 | 小 |
| **P1** | 新 EC（家電量販店/Yahoo!/Qoo10 等）を**カレンダー+手入力のみ**で追加（`dataSource:"manual"`, `partnership:none`） | 商品情報連携なしなら提携不要・低リスクで価値。P0 の上に**レコード追加 + 資産**で実現。 | 小（EC ごと） |
| **P1** | §7 ランキングの表現ガード + 事実ベース集計（小さく開始） | 既存データだけで出せる。誤認表現の定数化/リント。 | 小〜中 |
| **P2** | §4 ASP フィード Provider（registry + 1 provider、提携済みのみ） | 提携・規約確認が前提。鍵秘匿・Route 経由。価格/画像は参考値。 | 中 |
| **P2** | §6 公開共有（オプトイン・ホワイトリスト射影・noindex・失効） | プライバシー設計を固めてから。集客に効く。 | 中 |
| **P3** | §5 CSV インポート（管理導線・認可・ドライラン） | 認可と運用が要る。コア純粋関数から段階実装。 | 中〜大 |
| **P3** | §8 B2B 別ライン（画面/導線） | 体験分離の判断とリソースが要る。足場（`line`/`kind`）は P0 で先行。 | 大 |
| **P3** | 海外マーケット（`region`/通貨/i18n） | 影響が広い。`region` の足場のみ P0 で先行、本実装は後。 | 大 |

## 10. Codex に実装させる最小スコープ（提案）

**1 依頼 = 1 テーマ。まず P0 の「土台」だけ**を最小で実装する。商品情報の新規取得は一切含めない
（提携前 EC の取得なし／スクレイピングなし）。以降の P1+ は土台が緑になってから別依頼。

### 推奨 MVP スコープ（P0 のみ・破壊的変更なし）

| テーマ | 内容 | 受け入れ |
| --- | --- | --- |
| **M1: マスタ拡張（型）** | `Merchant` に §2.1 の任意フィールド（`kind`/`line`/`slug`/`urlHosts`/`dataSource`/`partnership`/`region`）を**任意で**追加。既存シード（amazon/rakuten）に `urlHosts` を記載。 | 既存テスト緑。型の破壊なし。`MerchantRepository` interface 不変。 |
| **M2: H2 urlHosts 駆動化** | `detectMerchantIdFromUrl` を `merchant.urlHosts` 優先で突合（コードの `knownHosts` はフォールバック据置）。 | amazon/rakuten で従来と同一判定（回帰テスト追加）。 |
| **M3: H1 colorToken 駆動化** | `merchantToneClasses` を 1 か所の静的マップへ整理 + safelist 文書化。**未知トークンは中立 fallback**（既存維持）。 | 未知 colorToken でも壊れず中立配色（テスト）。 |
| **M4: capability 純粋関数** | `getMerchantCapabilities(merchant, env)`（§3.2）を**純粋関数**で新規。既定最小権限（`partnership` 無し→商品情報系なし）。 | 単体テスト: 提携なし→`calendar`/`manualWishlist` のみ。`approved`+scope+flag で各 capability 付与。 |

- **含めない**（別依頼に回す）: ASP/CSV/公開共有/ランキング/B2B 画面/海外。新 EC の**実レコード追加**も M1〜M4 が
  緑になってから（資産・提携確認とセットで）。
- **H3（affiliate registry）**は影響範囲がやや広いので、M1〜M4 の次の単独依頼として切る（楽天変換は 1 provider
  として登録、シグネチャ不変）。

### Codex への引き継ぎ時の注意（既存ルール）

- Claude と**同時編集しない**。1 依頼 = 1 テーマ。実装後に Claude が確認。
- 変更後 `npm run lint` / `npm run test` / `npm run build` を実行し、失敗は原因と対処を記録。
- 既存テスト（`merchant`/`affiliate`/`price`/`wish-item`/`calendar`/`history`/`share`/`reminder` 等）を緑に保つ。
- 秘密情報・提携前 EC の商品情報を**コミット/取得しない**。

## 11. テスト観点（設計時点・実装時に具体化）

純粋関数（単体・必須）:
- `detectMerchantIdFromUrl`（urlHosts 駆動後）: マスタ `urlHosts` で突合、サブドメイン一致、未知ホスト→null、
  amazon/rakuten 回帰。
- `getMerchantToneClass`: 既知トークン→対応クラス、**未知トークン→中立 fallback**（追加忘れでも壊れない）。
- `getMerchantCapabilities`: 提携なし→`calendar`/`manualWishlist` のみ。`approved`+scope+env flag の組合せで
  `image`/`priceSuggestion`/`affiliate`/`feedImport` を境界含めて検証。env flag false で商品情報系が付かない。
- （ASP/CSV 実装時）`mapAspFeedItem` / CSV `sanitize`・`validate`: **許可ホスト外画像→null**、
  CSV インジェクション無害化、上限超過 reject、未 approved merchantId reject。
- （公開共有実装時）公開射影: `userId`/メモ/メール/追跡パラメータが**ペイロードに含まれない**こと（混入防止）。

回帰:
- 既存の amazon/rakuten 表示・カレンダー・フィルタ・アフィリエイト変換が従来どおり。
- `npm run lint` / `npm run test` / `npm run build` 緑。
