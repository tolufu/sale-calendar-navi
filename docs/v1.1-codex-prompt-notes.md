## 2. v1.1 特有の落とし穴（必ず伝える）

1. **元URLの上書き廃止**
   現状 `WishlistForm` は楽天変換後URLで `productUrl` を上書きしている。v1.1 では
   `candidates[].originalUrl`（元）と `affiliateUrl`（変換後）を**分離保存**する。元URLを潰さない。

2. **真のソースは `candidates`、既存フィールドはミラー**
   `merchantId/productUrl/actualPriceMemo/lastCheckedAt` は `candidates[0]` から導出して同期する
   （設計 §5.3）。二重管理で値がズレないよう、保存直前に一括同期する1関数に集約する。

3. **後方互換は遅延移行で**
   v1 の localStorage データを読めなくしない。読み出し時に `migrateWishItem` を必ず通す。
   バルク書き換え（全件再保存）はしない。移行は冪等であること。

4. **実質価格計算は新関数を追加**
   既存 `calculateEffectivePrice` は消さない（テスト維持）。5項目仕様は `computeEffectivePrice` を新設。
   計算式は厳密に「商品価格 + 送料 - クーポン値引き - 付与ポイント × ポイント換算率」。`pointRate` 既定 1.0、
   結果は整数丸め＆0未満は0。`productPrice=null` は null（差額対象外）。

5. **差額・希望価格表示の文言**
   「最安」「最安保証」「お得」等の断定語は使わない。必ず「手入力値であり正確性を保証しない」注記を添える。
   差額は「{EC}が {¥X} 低い見込み（手入力値）」のような中立表現。

6. **AF変換は merchants マスタ駆動・関数化**
   provider/enabled を `merchants[id].affiliate` から参照。enum 固定にしない。ダミーパラメータで可だが
   `buildAffiliateUrl` の中だけ差し替えれば実IDロジックに移れる構造に。冪等（二重変換しない）。

7. **参考リンクは保存だけ**
   価格.com 等のURLは「開くためのリンク」を保存するのみ。価格/在庫/画像の取得・転載・埋め込みは禁止。
   URLは http/https のみ・parse 可能のみ許可。

8. **ロジックはユーティリティに切り出してテスト**
   差額算出・移行・バリデーションを UI 内にベタ書きしない。`computeEffectivePrice` / `pickEffectivePriceDiff` /
   `migrateWishItem` / 参考リンクバリデーションは純粋関数化して単体テスト必須。

9. **画像方針据え置き**
   商品画像はプレースホルダーのみ。画像URL入力欄は作らない（v1〜v1.2方針）。
