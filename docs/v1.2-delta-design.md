## 0. v1.2 のゴールと非ゴール

### ゴール（今回入れる）
- 月別SEO記事・セール攻略記事を**5本以上**用意できる記事基盤（一覧の抜粋表示・カテゴリ・月別導線）。
- トップ / セール詳細 / 記事 / サイドバーの**広告プレースホルダー**を配置種別（slot）で整理。
- 通知設定UIの拡張：メールアドレス任意登録 / 通知ON-OFF / **3日前・前日・開始時**のタイミング設定。
  実メール送信は**土台（型・保存・送信インターフェイス）のみ**。本番送信はしない。
- 保存商品のECと一致するセールの**リマインダー候補生成**（純粋関数・ローカル算出）。
- Xシェア：セール情報シェア / 価格比較結果シェア（**ユーザー確認後のみ**）。
  共有文面に**保存URL・匿名ID・非公開メモを含めない**。
- OGP / canonical / metadata の整備（記事・セール詳細・主要ページ）。
- 法的ページ：プライバシーポリシー / 利用規約 / 運営者情報 / 問い合わせ導線。
- 履歴タブの完成：最近チェックしたセール・記事を最大30件、個別削除・全削除。
- AdSense 申請トリガーの明文化。

### 非ゴール（今回やらない / 禁止）
- 実メールの**本番送信運用**（SMTP/SendGrid 等の実接続・cron 配信）。土台のみ。
- 外部ECの価格・在庫・画像・レビューの自動取得（スクレイピング禁止・据え置き）。
- 商品画像の外部取得（v1〜v1.2はプレースホルダーのみ）。
- 「最安」「最安保証」等の断定・保証表現、公式サービスと誤認されるロゴ・表現。
- 広告コードの**本番IDの仮埋め込み**（プレースホルダーのみ）。
- 買い時スコア本実装（v2）。

## 2. 記事データ構造とURL設計

### 2.1 型拡張（`src/lib/repositories/types.ts`）
既存 `Article` を**後方互換**で拡張（既存フィールドは削除しない）:

```ts
export type ArticleCategory = "monthly" | "strategy" | "guide";

export type Article = {
  slug: string;            // URL用。半角英数とハイフンのみ（後述の規約）
  title: string;
  body: string;
  ogImage: string;         // プレースホルダーSVGのみ（外部画像禁止）
  tags: string[];
  publishedAt: string;     // ISO（+09:00）
  // ===== v1.2 追加（任意） =====
  excerpt?: string;        // 一覧・metadata description 用の抜粋（〜120字目安）
  category?: ArticleCategory; // 未設定は "guide" 扱い
  month?: string;          // 月別記事のみ "YYYY-MM"（例 "2026-06"）。それ以外は undefined
  updatedAt?: string;      // 改稿日（任意）
};
```

- `excerpt` 未設定時は `body` 先頭段落を切り詰めて代用するヘルパ（`getArticleExcerpt(article, len=120)`）を `src/lib/utils/article.ts`（新）に置く。
- 記事本文は Markdown ではなく現行の「`\n\n` 区切りプレーンテキスト」を踏襲（パーサ追加は非ゴール）。

### 2.2 URL設計
- 詳細: 現行どおり `/articles/[slug]`（変更しない）。
- 一覧: `/articles`（カテゴリ・月でクライアントフィルタ。新ルートは増やさない）。
- slug 規約: `^[a-z0-9-]+$`。月別記事は `YYYY-MM-<topic>`（例 `2026-06-sale-strategy`）。
  攻略記事は `<topic>-guide`（例 `point-cycle-guide`）。
- 一覧のフィルタは**クエリパラメータ**で表現（共有・SEO 配慮）: `?category=monthly` / `?month=2026-06`。
  動的セグメント増設はしない（静的データで十分・ビルド簡素）。

### 2.3 必要本数と内容方針（AdSense トリガーと連動 → §10）
- 最低 **5本**（うち月別 2本以上を推奨）。例:
  1. `2026-06-sale-strategy` … 6月セールの見直しポイント（monthly）
  2. `2026-07-sale-strategy` … 7月セールの見直しポイント（monthly）
  3. `point-cycle-guide` … ポイント施策の考え方（strategy）
  4. `coupon-memo-guide` … クーポンとメモの整理（strategy）
  5. `monthly-sale-prep`（既存）… 買い物メモの作り方（guide）
- 内容は**一次情報（自分のメモ運用ノウハウ）**で構成。外部価格・在庫・画像の転載や「最安保証」表現は禁止。
- 各記事に `excerpt` を必ず付与（一覧・metadata の品質確保）。

## 3. Metadata / OGP / canonical 方針

### 3.1 共通方針
- `metadataBase` は既存（`NEXT_PUBLIC_SITE_URL` ?? localhost）を継続。**本番URLは環境変数**で与える。
- 各ページで `alternates.canonical` を**相対パス**で指定（`metadataBase` 起点で絶対化される）。
- `description` は 50〜120字目安。本文全文の流し込みは禁止（記事は `excerpt` を使う）。
- OG 画像はプレースホルダー（`/images/placeholders/og-sale-calendar.svg`）。記事個別画像も自作SVGのみ。

### 3.2 ページ別
| ページ | title | description | canonical | OG |
| --- | --- | --- | --- | --- |
| トップ `/` | 既定 | サービス説明 | `/` | 既定OG |
| カレンダー `/calendar` | "セールカレンダー" | 月別セール予定の確認 | `/calendar` | 既定OG |
| 記事一覧 `/articles` | "記事一覧" | セール準備・攻略の記事 | `/articles` | 既定OG |
| 記事詳細 `/articles/[slug]` | 記事タイトル | **`excerpt`**（なければ本文先頭切詰め） | `/articles/${slug}` | `article.ogImage` |
| セール詳細 `/sales/[id]` | セールタイトル | セール概要（`description` 先頭切詰め） | `/sales/${id}` | 既定OG |
| 法的ページ | 各ページ名 | 各ページ概要 | 各パス | 既定OG |

- セール詳細は現状 `generateMetadata` が無い → **追加**（`saleEvents` から取得、無ければ汎用文言）。
- 記事詳細の `generateMetadata` は `description` を `excerpt` ベースに修正し `alternates.canonical` を追加。
- 構造化データ（JSON-LD）は任意（入れるなら記事に `Article`/`BreadcrumbList`）。v1.2 では**任意・後追い可**と明記。

## 4. 広告プレースホルダー配置

### 4.1 コンポーネント拡張（`src/components/ui/AdPlaceholder.tsx`）
現状は固定文言。**slot 種別とラベル**を受ける形へ（本番コードは入れない）:

```ts
type AdSlot = "top" | "sidebar" | "article" | "sale-detail" | "inline";

export function AdPlaceholder({
  slot = "inline",
  className,
}: { slot?: AdSlot; className?: string }): JSX.Element;
```

- 表示は破線枠＋「広告枠（未設定）」を維持。`slot` ごとに最小高さ/aspect を変える程度（CLS 抑制）。
- **本番のAdSenseコード・本番IDは埋め込まない**。将来差し替え位置として `data-ad-slot={slot}` 等の目印のみ。
- `aria-hidden` ではなく `role="complementary"` / ラベル付きで、広告枠であることを明示。

### 4.2 配置
| 画面 | 位置 | slot |
| --- | --- | --- |
| トップ `/` | ファーストビュー下／コンテンツ間 1枠 | `top` |
| セール詳細 | 既存の本文下（維持） | `sale-detail` |
| 記事詳細 | 本文下（維持）＋サイドバー | `article` / `sidebar` |
| 記事一覧 | 一覧の途中に1枠（任意） | `inline` |

- 広告枠は**1画面あたり過剰に置かない**（UX・将来のポリシー配慮）。各画面 1〜2枠まで。
- 広告とコンテンツの誤認を避けるため、必ず「広告」ラベルを枠内に残す。

## 5. 通知設定データ構造

### 5.1 型拡張（`src/lib/repositories/types.ts`）
`NotificationSetting` を**後方互換**で拡張（`leadDays` は残す）:

```ts
export type NotificationTimings = {
  threeDaysBefore: boolean; // 3日前
  dayBefore: boolean;       // 前日
  onStart: boolean;         // 開始時
};

export type NotificationSetting = {
  userId: string;
  enabled: boolean;
  leadDays: number;                 // 互換維持（既存UI・既存保存データ用）
  perMerchant: Record<string, boolean> | null;
  // ===== v1.2 追加（任意） =====
  email?: string | null;            // 任意登録。未入力は null
  timings?: NotificationTimings;    // 未設定は既定 {3日前:true, 前日:true, 開始時:false} 扱い
  emailVerified?: boolean;          // 土台のみ。既定 false（実検証フローは非ゴール）
};

export const DEFAULT_TIMINGS: NotificationTimings = {
  threeDaysBefore: true, dayBefore: true, onStart: false,
};
```

### 5.2 移行・正規化
- `src/lib/utils/notification.ts`（新）に `normalizeNotificationSetting(raw): NotificationSetting`。
  - `timings` 未設定 → `leadDays` から推定して補完（`leadDays>=3`→3日前ON、`>=1`→前日ON など）か、
    単純に `DEFAULT_TIMINGS` を入れる（**実装はどちらか一方に統一**。設計推奨は `DEFAULT_TIMINGS`）。
  - `email` 未設定 → null。空文字はトリム後 null 化。
- `LocalNotificationRepository.get` の読み出し直後に正規化を通す（既存データを壊さない）。

### 5.3 メールバリデーション
- `email` は任意。入力時のみ `^[^\s@]+@[^\s@]+\.[^\s@]+$` 程度の軽量チェック（厳密RFCは不要）。
- バリデーションは `src/lib/utils/notification.ts` の `isValidNotificationEmail(email)` に切り出してテスト。
- **重要**: メールアドレスは**機密相当**。ログ出力・共有文面・履歴・URLクエリに含めない。localStorage 保存のみ。

### 5.4 UI（`NotificationSettings.tsx`）
- 通知 ON-OFF（既存）/ メール入力（任意・形式エラー表示）/ タイミング3チェックボックス。
- `enabled=false` のときタイミング・メールは入力可だが「現在通知はオフです」を明示。
- 保存は既存 `notifications.save` を流用（型拡張のみ）。Toast で完了表示。
- 「実メール送信は今後対応予定（現在は設定の保存のみ）」の注記を出す（過度な期待を与えない）。

## 6. リマインダー生成ロジック

### 6.1 目的と原則
- 保存商品（`WishItem`）のECと**今後のセール**（`SaleEvent`）を突合し、見直しタイミングの候補を**ローカルで算出**。
- 実メール送信はしない（候補生成と画面表示まで）。純粋関数化してテスト。

### 6.2 配置と型（`src/lib/utils/reminder.ts`（新））
```ts
export type ReminderTimingKind = "threeDaysBefore" | "dayBefore" | "onStart";

export type ReminderCandidate = {
  saleEventId: string;
  saleTitle: string;
  merchantId: string;
  wishItemIds: string[];     // このセールECに一致する保存商品の id 群
  timing: ReminderTimingKind;
  notifyAt: string;          // ISO。算出した通知予定時刻
};

export function generateReminderCandidates(args: {
  now: Date;
  sales: SaleEvent[];
  wishItems: WishItem[];
  setting: NotificationSetting;
}): ReminderCandidate[];
```

### 6.3 アルゴリズム
1. `setting.enabled === false` → 空配列。
2. 対象セール = `sales` のうち `startAt > now`（既に開始済みは除外。開始時通知のみ「開始直前」を許容）。
3. EC突合 = 各セールについて、`wishItems` から
   - `item.merchantId === sale.merchantId`、**または** `item.candidates?.some(c => c.merchantId === sale.merchantId)`
   - に一致する item を集める。`perMerchant` が指定され当該 merchant が false なら除外。
   - 一致 0 件のセールは候補化しない。
4. タイミング展開 = `setting.timings`（無ければ `DEFAULT_TIMINGS`）で ON のものだけ:
   - `threeDaysBefore` → `notifyAt = startAt - 3日`
   - `dayBefore` → `notifyAt = startAt - 1日`
   - `onStart` → `notifyAt = startAt`
   - `notifyAt <= now` の候補は除外（過去分は出さない）。
5. 並び順 = `notifyAt` 昇順。同時刻は `saleTitle` で安定化。
6. 純粋関数（`now` を引数で受け、`Date.now()` を内部で呼ばない＝テスト容易）。

### 6.4 表示（任意・通知設定画面 or トップ）
- 「リマインダー候補」セクションとして `notifyAt` と対象セール・該当商品件数を表示。
- 表示文言は中立（「6/1 開始のセール前に 2件の保存商品を見直し」程度）。断定・保証表現は使わない。
- **保存URL・匿名ID・非公開メモは表示にもシェアにも出さない**（件数とタイトルのみ）。

## 7. Xシェア文面生成ロジックとプライバシー制約

### 7.1 配置と型（`src/lib/utils/share.ts`（新））
現状は各コンポーネントで `intent/tweet?text=` を直書き。**文面生成を関数化**してテスト可能にする。

```ts
export type ShareKind = "sale" | "priceComparison";

export function buildSaleShareText(sale: SaleEvent): string;
export function buildPriceComparisonShareText(args: {
  itemTitle: string;
  merchants: { name: string; effectivePrice: number | null }[];
}): string;

export function buildShareUrl(text: string, pageUrl: string): string; // intent/tweet を組み立て
```

### 7.2 文面方針
- セールシェア例: `「{セール名}」をセールカレンダー比較ナビでチェック中` + ハッシュタグ `#セールカレンダー比較ナビ`。
- 価格比較シェア例: `「{商品名}」の実質価格メモ（手入力）：{EC名} ¥X / {EC名} ¥Y ※手入力値・最新は各サイトで確認`。
  - **必ず「手入力値」「最新は各サイトで確認」の注記**を含める。「最安」「保証」は使わない。
- 共有URLには**自サイトの公開ページURL**（セール詳細 `/sales/[id]` 等）のみを付与。

### 7.3 プライバシー制約（禁止事項の徹底）
- 共有文面・共有URLに以下を**絶対に含めない**:
  - 保存した商品URL（`originalUrl`/`affiliateUrl`/`productUrl`）。
  - 匿名ID（`userId`）。
  - 非公開メモ（`actualPriceMemo`/`note`/`priceMemo`/参考リンク）。
- 価格比較シェアは**ユーザーの明示確認後のみ**送信導線を出す:
  - ボタン → 確認モーダル（共有される文面プレビューを表示）→ 同意で `intent/tweet` を開く。
  - 既定では価格比較シェアボタンは「内容を確認してシェア」とし、ワンタップ送信にしない。
- セール情報シェアは公開情報のみのため確認モーダルは任意（プレビュー表示は推奨）。
- 文面生成関数は**入力に上記機密フィールドを受け取らない**シグネチャにする（誤混入を型で防ぐ）。

## 8. 履歴タブの完成仕様

### 8.1 記録対象とトリガー
- `type: "sale"` … セール詳細表示時に記録（**実装済み**）。
- `type: "article"` … 記事詳細表示時に記録（**新規追加**）。`ArticleDetail` 表示時に `history.create`。
  - 記事はサーバーコンポーネント由来のため、記録は**クライアント側**（小さな `"use client"` フックか、
    既存パターンに合わせ詳細を client 化）で行う。匿名IDは履歴に保存してよい（ローカル）が共有はしない。
- `type: "deletedWish"` … 欲しいもの削除時に記録（任意・既存タブに存在）。今回必須化はしない。

### 8.2 一覧仕様
- 上限 **30件**（`HISTORY_LIMIT=30` 既存）。`occurredAt` 降順（`limitHistoryItems` 既存）。
- フィルタタブ: すべて / セール / 記事 / 削除済み（既存）。**「最近チェックした」**の主旨に合わせ
  ラベルを「セール」「記事」に確定。
- 各行: タイトル / 日時 / 種別 / `href` があれば「再訪」/ 「削除」。
- 個別削除（`remove`）・全削除（`clear`）は**実装済み**。全削除には確認（`window.confirm` か簡易モーダル）を追加。

### 8.3 重複・空表示
- 同一 `href` の連続表示で履歴が膨れないよう、`create` 時に**直近同一hrefは更新（occurredAt 更新）**するか、
  単純追記のままにするかを選ぶ。設計推奨: **同一 `type+href` は最新1件に集約**（`upsertHistory` ヘルパを
  `src/lib/utils/history.ts` に追加してテスト）。
- 空表示文言を更新（現状「v1.2 で…拡張します」→「まだ履歴がありません。セールや記事を見ると記録されます」）。

## 9. 法的ページの最低限の文面方針

新規ルート（静的・サーバーコンポーネントで可）:
- `/privacy`（プライバシーポリシー）
- `/terms`（利用規約）
- `/about`（運営者情報）
- `/contact`（問い合わせ導線）

`SiteFooter` に4ページ＋既存注記へのリンクを追加。

### 9.1 各ページの最低限の論点（断定や法務助言ではなく事実ベースで記述）
- **プライバシーポリシー** `/privacy`:
  - 収集する情報: 匿名認証ID、ローカル保存データ（欲しいもの・履歴・通知設定）、任意のメールアドレス。

## 10. AdSense 申請トリガー（明文化）

以下が**すべて揃った時点**で AdSense 申請可能とする（申請前チェックリスト）:

1. **月別記事を含む記事が 5本以上**公開されている（§2.3。うち月別 2本以上を推奨）。
2. **プライバシーポリシー** `/privacy` 公開（Cookie・広告・データ削除の記載を含む）。
3. **運営者情報** `/about` 公開（運営主体と連絡手段）。
4. **利用規約** `/terms` 公開。
5. **問い合わせ導線** `/contact` 公開（`mailto:` か外部フォームで可）。

補足:
- 申請まで**本番の広告コード・本番IDは入れない**（プレースホルダーのまま）。承認後に差し替える。
- フッターから上記4ページへ常時アクセスできること。
- 記事は実質的な内容（一次情報）であること（自動生成・薄いコンテンツは避ける）。

## 11. テスト観点

ユーティリティ（単体・必須）:
- `getArticleExcerpt`: `excerpt` 優先、無ければ本文先頭を指定長で切詰め、改行/空白の正規化。
- `normalizeNotificationSetting`: `timings`/`email` 未設定の補完、空文字メールの null 化、冪等。
- `isValidNotificationEmail`: 正常/異常（@なし・空白混入・空文字）。
- `generateReminderCandidates`:
  - `enabled=false` で空。
  - EC一致（`merchantId` 直一致 / `candidates[].merchantId` 一致）で候補化、不一致は除外。
  - `perMerchant` で当該 merchant false のとき除外。
  - タイミング ON/OFF の展開、過去 `notifyAt` の除外、`now` 境界。
  - 並び順（`notifyAt` 昇順・安定）。
- `buildSaleShareText` / `buildPriceComparisonShareText` / `buildShareUrl`:
  - 文面に注記（手入力値・各サイト確認）が含まれる。
  - **保存URL・userId・非公開メモが文面/URLに混入しない**（混入防止の回帰テストを必ず置く）。
  - `buildShareUrl` のエンコード。
- `upsertHistory`（採用時）: 同一 `type+href` の集約・最新化、上限30の維持。

コンポーネント:
- 通知設定: メール形式エラー表示、タイミング保存の永続化、`enabled=false` 時の注記。
- 記事一覧: 抜粋表示、カテゴリ/月フィルタ（クエリ）で絞り込み。
- 履歴: 記事閲覧で `article` 履歴が増える、個別削除/全削除（確認）、空表示文言。
- 価格比較シェア: 確認モーダルを経由しないと送信導線が開かない。
- 広告: 各 slot がラベル付きで描画され、本番IDが含まれない。

メタ/回帰:
- 記事詳細・セール詳細の `generateMetadata` に canonical が入る、description が抜粋ベース。
- 既存テスト（`affiliate`/`price`/`wish-item`/`history` 等）が緑のまま。
- `npm run lint` / `npm run test` / `npm run build` 緑。
- 後方互換: 旧 `NotificationSetting`（`email/timings` なし）・旧履歴データを読んでも例外なく描画。
