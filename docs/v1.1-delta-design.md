## 0. v1.1 のゴールと非ゴール

### ゴール（今回入れる）
- 希望価格の表示・編集の強化（希望価格内かどうかの中立バッジ）。
- 現在の実質価格メモ（自由記述, 既存 `actualPriceMemo` を継続利用）。
- 詳細価格入力の折りたたみを**構造化**：商品価格 / 送料 / クーポン値引き / 付与ポイント / ポイント換算率。
- 実質価格の自動計算：`商品価格 + 送料 - クーポン値引き - 付与ポイント × ポイント換算率`。
- Amazon候補・楽天候補の実質価格があれば**差額表示**（手入力値である旨の注記つき）。
- 参考比較リンク（価格.com / メーカー公式 / その他URL）。**リンク保存のみ。価格取得はしない。**
- 楽天URLのアフィリエイトリンク変換：**元URLと変換後URLの両方を保持**。ダミーパラメータでよいが関数化。
- 最終確認日の表示。
- データ移行・後方互換（v1データを壊さない）。

### 非ゴール（今回やらない / 禁止）
- 楽天APIなど外部APIの連携。
- 価格.com 等からの価格・在庫・画像の自動取得（スクレイピング禁止）。
- 外部サイト価格を「最安値」「最安保証」として表示すること（断定・保証表現は不可）。
- 買い時スコア本実装（v2）、広告枠（v1.2）、外部商品画像の取得（v1〜v1.2はプレースホルダーのみ）。

## 1. 現行のデータ型・保存形式（確認結果）

### 1.1 型（`src/lib/repositories/types.ts`）
現行 `WishItem` は **1アイテム=1EC=1URL** 構造。価格は `desiredPrice`（数値）と
`actualPriceMemo`（自由記述）のみで、内訳の構造化フィールドは持たない。

```ts
type WishItem = {
  id: string; userId: string; title: string;
  productUrl: string;          // 保存時に楽天は変換後URLで上書きされている（元URL未保持）
  merchantId: string;
  desiredPrice: number | null;
  actualPriceMemo: string | null;
  targetSaleEventId: string | null;
  placeholderKey: string;
  note: string | null;
  createdAt: string; updatedAt: string;
};
type WishItemInput = Omit<WishItem, "id" | "userId" | "createdAt" | "updatedAt">;
```

未使用だが `Offer` 型（`merchantId / productGroupId / productUrl / priceMemo / updatedAt`）が定義済み。
v1.1 の「EC別候補」と概念が近いが、`productGroupId` 前提のため**今回は流用せず**、後述の `PriceCandidate` を新設する。

### 1.2 保存形式
- localStorage キー: `sale-calendar-navi:${userId}:wishlist` に `WishItem[]` を JSON 配列で保存
  （`src/lib/repositories/local-storage.ts`）。
- Firestore 実装は現状 `createFirestoreRepositories()` が local 実装にフォールバック（`firestore.ts`）。
  本書では将来の Firestore 形 `users/{uid}/wishItems/{itemId}` を併記する。

### 1.3 既存の関連ロジック
- `convertRakutenAffiliateUrl(url, affiliateId?)`（`src/lib/utils/affiliate.ts`）: 楽天URL かつ ID あり時のみ
  `hb.afl.rakuten.co.jp/hgc/g00dummy/` 形式へ変換。**string を返すのみで元URLは保持しない。**
- `calculateEffectivePrice(basePrice, discount, pointValue)`（`src/lib/utils/price.ts`）: 引数3つの素朴計算。
  今回の5項目仕様には不足のため**新関数を追加**する（後述 §6）。
- `WishlistForm.tsx`: 保存時に `productUrl` を変換後URLで上書き（元URL喪失）。詳細折りたたみは
  「実質価格メモ / 対象セール / 補足メモ」のみ。
- `WishlistList.tsx`: 希望価格・登録日・前回メモ・外部リンク・関連セールを表示。インライン編集は
  title / desiredPrice / actualPriceMemo のみ。

> **重要な現状課題**: 楽天変換後URLで `productUrl` を上書きしているため、元URLが残らず再変換・検証ができない。
> v1.1 では元URLと変換後URLを分離して保持する（§7）。

## 2. 追加・変更する型

`src/lib/repositories/types.ts` に以下を追加。**既存フィールドは削除しない**（後方互換）。

```ts
/** 詳細価格の内訳（すべて任意。null=未入力） */
export type PriceBreakdown = {
  productPrice: number | null;   // 商品価格
  shippingFee: number | null;    // 送料
  couponDiscount: number | null; // クーポン値引き
  grantedPoints: number | null;  // 付与ポイント（ポイント数）
  pointRate: number | null;      // ポイント換算率（0〜1。null時は既定1.0=1pt=1円）
};

/** 参考比較リンク（保存のみ。価格取得はしない） */
export type ReferenceLink = {
  id: string;
  kind: "kakaku" | "maker" | "other"; // 価格.com / メーカー公式 / その他
  label: string;                       // 表示名（必須・最大40字目安）
  url: string;                         // http/https のみ
};

/** EC別の価格候補（Amazon候補・楽天候補など） */
export type PriceCandidate = {
  merchantId: string;
  originalUrl: string;          // 入力された元URL（必須）
  affiliateUrl: string | null;  // 変換後URL（楽天等。未変換時はnull）
  breakdown: PriceBreakdown;    // 内訳
  priceMemo: string | null;     // 実質価格メモ（自由記述）
  lastCheckedAt: string | null; // この候補の最終確認日（ISO）
};

export const WISH_ITEM_SCHEMA_VERSION = 2;
```

`WishItem` の拡張（追加分のみ。既存はそのまま残す）:

```ts
export type WishItem = {
  // ===== 既存（v1, 後方互換のため維持） =====
  id: string; userId: string; title: string;
  productUrl: string;             // = 主候補の affiliateUrl ?? originalUrl のミラー
  merchantId: string;             // = 主候補(candidates[0])の merchantId のミラー
  desiredPrice: number | null;
  actualPriceMemo: string | null; // = 主候補 priceMemo のミラー（自由記述の互換窓口）
  targetSaleEventId: string | null;
  placeholderKey: string;
  note: string | null;
  createdAt: string; updatedAt: string;

  // ===== v1.1 追加 =====
  schemaVersion?: number;             // 移行管理（新規=2, 旧データ=未設定→1扱い）
  candidates?: PriceCandidate[];      // EC別候補。candidates[0] を主候補とする
  referenceLinks?: ReferenceLink[];   // 参考比較リンク
  lastCheckedAt?: string | null;      // アイテム単位の最終確認日（候補横断の最新）
};
```

### 設計判断（なぜ候補配列にするか）
- 要件「Amazon候補・楽天候補の実質価格があれば差額表示」は**1アイテムで複数ECの内訳を持つ**ことを要求する。
- 既存の単一 `merchantId/productUrl` は **`candidates[0]`（主候補）のミラー**として維持し、
  既存UI・既存読み出し・エクスポートが壊れないようにする（ミラーは書き込み時に同期, §5.3）。
- これにより「最小差分」かつ「将来 EC を増やせる」設計を両立する。
- 互換重視で段階導入したい場合の縮小案は §10 に記載。

- パス `users/{uid}/wishItems/{itemId}`。`candidates` / `referenceLinks` は**マップの配列**として保存可

- 読み出し側で localStorage と同じ正規化関数を共有する（実装が分かれても移行ロジックは1本に）。

## 4. 既存データ移行方針（後方互換）

**遅延移行（lazy migration）** を採用。バルク書き換えはしない。

正規化関数を新設（`src/lib/utils/wish-item.ts` 想定）:
```ts
export function migrateWishItem(raw: WishItem): WishItem
```
仕様:
1. `schemaVersion >= 2` かつ `candidates` が存在 → そのまま返す（冪等）。
2. それ以外（v1データ）→ 主候補1件を合成して v2 化:
   - `candidates = [{ merchantId, originalUrl: productUrl, affiliateUrl: null,
     breakdown: 全null, priceMemo: actualPriceMemo, lastCheckedAt: null }]`
     - 注: v1 の `productUrl` は楽天では変換後URLの可能性がある。元URL確定不能のため
       `originalUrl` に現値を入れ、`affiliateUrl=null` とする（次回編集時にユーザーが上書き可能）。
   - `referenceLinks = []`, `lastCheckedAt = null`, `schemaVersion = 2`。
   - 既存 `productUrl/merchantId/desiredPrice/actualPriceMemo` は**保持**（ミラーとして整合）。
3. 破壊しない：未知フィールドは温存。`breakdown` 欠損キーは null 補完。

適用箇所:
- `LocalWishlistRepository.list/get` の読み出し直後に `migrateWishItem` を通す。
- 書き込み（`create/update`）時に v2 形へ正規化して保存（読めた時に自然に v2 化される）。
- Firestore 実装でも同じ関数を通す。

> 移行の単体テスト必須（§9）。「v1 JSON を読み込んでも例外なく v2 として表示できる」を担保する。

## 5. リポジトリ層の変更

### 5.1 型
- `WishItemInput = Omit<WishItem, "id"|"userId"|"createdAt"|"updatedAt">` のまま。
  追加フィールド（`schemaVersion/candidates/referenceLinks/lastCheckedAt`）も Input に含まれる。
  - `schemaVersion` は Input 必須にせず、`create/update` 内で常に `WISH_ITEM_SCHEMA_VERSION` を付与する。

### 5.2 メソッド
- 既存の `list/get/create/update/remove` のシグネチャは**変更しない**。
- `create/update` 内部で:
  1. ミラー同期（§5.3）。
  2. `schemaVersion = 2`、`lastCheckedAt` 再計算（候補の最大）。

### 5.3 ミラー同期ルール（単一ソース化）
- **真のソース = `candidates`**。`candidates[0]` を主候補とする。
- 保存前に以下を主候補から導出して上書き:
  - `merchantId = candidates[0].merchantId`
  - `productUrl = candidates[0].affiliateUrl ?? candidates[0].originalUrl`
  - `actualPriceMemo = candidates[0].priceMemo`
  - `lastCheckedAt = candidates の lastCheckedAt の最大（なければ null）`
- これにより、v1 を前提にした表示・エクスポートが引き続き動く。
- フォームが候補を1件しか作らない場合も主候補のみで成立する。

## 6. 実質価格計算

新関数を `src/lib/utils/price.ts` に追加（既存 `calculateEffectivePrice` は当面残す／テスト維持）。

```ts
/** 実質価格 = 商品価格 + 送料 - クーポン値引き - 付与ポイント × ポイント換算率
 *  内訳が不足（商品価格が null）なら null を返す。 */
export function computeEffectivePrice(b: PriceBreakdown): number | null;
```

仕様:
- `productPrice == null` → `null`（計算不能＝差額表示の対象外）。
- それ以外は各 null を 0 として:
  `effective = productPrice + shippingFee - couponDiscount - grantedPoints * (pointRate ?? 1)`
- `pointRate` 既定は 1.0（1pt=1円）。0〜1 の範囲を想定（例: 0.5 など将来用）。負値は入力バリデーションで弾く。
- 結果は `Math.round` で整数化し、`Math.max(0, …)` で 0 未満を 0 に丸める。
- 純粋関数（副作用なし）。表示は `formatPrice` を流用。

### 差額表示ロジック（Amazon候補 / 楽天候補）
- 各候補の `computeEffectivePrice` を算出 → **null を除外**。
- 有効候補が 2 件以上のとき、最小値と次点の差 `diff = second - min` を計算。
- 表示文言（**断定・最安表現を避ける**）:
  - 例:「手入力の実質価格では {merchant名} が {¥diff} 低い見込み」
  - 必ず注記:「価格はあなたの手入力値です。最安・正確性を保証するものではありません。」
- 有効候補が 1 件以下 → 差額は出さず、その候補の実質価格のみ表示。
- 同額 → 「実質価格は同程度（手入力値）」と中立表記。
- ロジックは `pickEffectivePriceDiff(candidates): { minMerchantId, diff } | null` 等のユーティリティに切り出してテスト。

### 希望価格の表示・編集強化
- 主候補（または最小）の実質価格が `desiredPrice` 以下なら中立バッジ「希望価格内」を表示。
  超過なら「希望価格まで {¥差}」。`desiredPrice` 未設定なら非表示。
- 「お得」「最安」等の断定語は使わない。

## 7. 楽天AF変換関数の仕様（元URL+変換後URLを両方保持）

既存 `convertRakutenAffiliateUrl` は string 返却で元URL喪失。v1.1 で**両保持 API** を追加する。

`src/lib/utils/affiliate.ts`:
```ts
export type AffiliateConversion = {
  originalUrl: string;
  affiliateUrl: string | null; // 変換できない場合は null
  converted: boolean;
  provider: string | null;     // "rakuten" 等。非対象は null
};

/** merchant.affiliate 設定駆動で変換。将来の実IDロジックに差し替えやすいよう関数化。 */
export function buildAffiliateUrl(
  originalUrl: string,
  options: { provider?: string | null; affiliateId?: string; enabled?: boolean }
): AffiliateConversion;
```

仕様:
- `provider !== "rakuten"` または `enabled === false` → `{ affiliateUrl: null, converted: false, provider }`。
- 楽天かつ `affiliateId` あり → ダミー形式 `hb.afl.rakuten.co.jp/hgc/g00dummy/?pc=<原URL>&link_type=hybrid_url&me_id=<id>` を生成
  （**ダミーで可。実IDロジックはこの関数の中だけ差し替え**できるようにする）。
- `affiliateId` 未設定 → `{ affiliateUrl: null, converted: false }`（元URLのまま保存）。AC方針どおり。
- ホスト判定は `rakuten.co.jp` 系のみ。非楽天URLは `converted: false`。
- **冪等性**: 入力が既に `hb.afl.rakuten.co.jp` の場合は二重変換しない（`affiliateUrl=入力, converted=true` として返す）。
- 元URL（クエリ/ハッシュ）を保持してエンコードする。URL parse 失敗時は `converted: false`。
- 既存 `convertRakutenAffiliateUrl` は内部で `buildAffiliateUrl` を呼ぶ薄いラッパに置換し、既存テスト・呼び出しを維持
  （フォームは新APIへ移行し、`candidates[].originalUrl/affiliateUrl` に両方を保存）。
- merchants マスタの `affiliate.provider/enabled` を参照（enum固定にしない）。

## 8. フォーム / 一覧カード UI 変更

### 8.1 登録・編集フォーム（`WishlistForm.tsx`）
基本領域（常時表示）:
- 商品名 / 商品URL（= 主候補 originalUrl）/ EC（自動判定継続）/ 希望価格。
- URL 入力時に `buildAffiliateUrl` を呼び、変換プレビュー（「楽天アフィリエイトリンクに変換して保存します」）を表示。

折りたたみ「詳細価格入力」（AC-008 維持・**構造化**）:
- 数値5項目: 商品価格 / 送料 / クーポン値引き / 付与ポイント / ポイント換算率（既定1.0）。
- **実質価格のライブプレビュー**（`computeEffectivePrice`）。計算不能時は「商品価格を入れると計算します」。
- 実質価格メモ（自由記述, 既存 `actualPriceMemo` 相当 = 主候補 priceMemo）。
- 最終確認日（日付入力。未入力なら保存時の当日を任意で補完するかは下記）。
- 対象セール / 補足メモ（既存）。

比較候補（任意・段階導入可）:
- 「比較候補を追加」で 2件目以降の候補（別ECのURL+内訳+メモ）を入力。v1.1 初期は Amazon/楽天の2枠で十分。
- 候補ごとに自動EC判定・AF変換を適用。

参考比較リンク:
- 行追加式（kind セレクト[価格.com/メーカー公式/その他] + 表示名 + URL）。最大5件目安。
- バリデーションは §8.3。

保存時:
- 候補配列を組み立て → `buildAffiliateUrl` 適用 → ミラー同期（§5.3）→ `create/update`。
- 楽天変換で `productUrl` を直接上書きする現挙動は廃止し、`originalUrl/affiliateUrl` 分離保存に変更。

### 8.2 一覧カード（`WishlistList.tsx`）
- 希望価格 + 希望価格内バッジ（§6）。
- 候補ごとの実質価格（EC名 + ¥）。2件以上あれば**差額行**（注記つき）。
- 最終確認日「最終確認: yyyy/mm/dd」（`lastCheckedAt`、無ければ非表示）。
- 参考比較リンク（外部リンクとして列挙。価格は出さない・取得しない）。
- 外部リンクは `affiliateUrl ?? originalUrl`（楽天は変換後を優先）。
- インライン編集の拡張: title / desiredPrice に加え、主候補の内訳5項目・メモを編集可能に
  （複雑になりすぎる場合は「詳細編集はフォーム画面へ」リンクに留める判断も可）。
- すべて `rel="noreferrer"`、`target="_blank"` 継続。

### 8.3 参考リンクのバリデーション
- `url`: `new URL()` で parse 可能、かつ `protocol` が `http:`/`https:` のみ。失敗はエラー表示し保存不可。
- `label`: 必須・トリム後1字以上・最大40字目安。
- `kind`: enum3値（`kakaku`/`maker`/`other`）。
- 件数上限（例5件）。空行は保存対象外。
- ソフトチェック（任意・ブロックしない）: `kind=kakaku` で host が `kakaku.com` でないとき注意文を出す程度。
- **禁止事項の徹底**: ここで保存するのは「ユーザーが開くためのURL」だけ。価格・在庫・画像の取得や転載はしない。

## 9. テスト観点

ユーティリティ（単体・必須）:
- `computeEffectivePrice`:
  - 全項目あり（例 13800+0-1000-1380×1=11420）。
  - `productPrice=null` → null。
  - `pointRate` 既定1.0 / 0.5 のケース。
  - 0未満→0クランプ、丸め。
- `buildAffiliateUrl`:
  - 楽天+ID → 変換され `originalUrl/affiliateUrl` 両方返る、`me_id` 反映。
  - ID無し → `affiliateUrl=null, converted=false`。
  - 非楽天 → 非変換。
  - 既に afl URL → 二重変換しない（冪等）。
  - クエリ/ハッシュ保持、parse失敗時の安全な戻り。
- `migrateWishItem`:
  - v1 JSON → v2（candidates[0] 合成・データ無損失）。
  - v2 入力で冪等。
  - `breakdown` 欠損キーの null 補完。
- 差額ロジック `pickEffectivePriceDiff`:
  - 2件で min/diff、同額、1件以下で null。
- 参考リンクバリデーション: 不正プロトコル拒否 / 正常許可 / 件数上限。

コンポーネント:
- フォーム: 詳細折りたたみ開閉、実質価格ライブプレビュー更新、保存で内訳が永続化。
- 一覧: 候補2件で差額行が出る・注記が出る、最終確認日表示、希望価格内バッジ。
- 後方互換: v1 形 localStorage を読み込んでも例外なく描画。

回帰:

- `npm run lint` / `npm run test` / `npm run build` 緑。

## 10. 段階導入オプション（縮小案）

差分を更に小さくしたい場合の代替（推奨はしないが選択肢）:
- **単一候補のみ先行**: `candidates` を導入せず `WishItem` に `breakdown` を1組だけ直付け＋
  `originalUrl/affiliateUrl/lastCheckedAt` を追加。差額表示（複数EC比較）は v1.1.x へ後送り。
- 利点: 移行・UIが軽い。欠点: 「Amazon候補・楽天候補の差額」を満たせず要件未達。
- 結論: 要件充足のため**本則（候補配列）を推奨**。比較候補のフォーム入力だけ後追い実装でも可
  （型・保存・移行・差額ロジックは先に入れておく）。
